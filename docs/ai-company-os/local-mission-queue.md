# AI Company OS: Local Mission Queue

This document specifies the local queue lifecycle, state transitions, worker emulation, and artifact contracts for simulated executions.

## Mission Lifecycle States

A mission processed by the router transitions through the following lifecycle states:

- `PLANNED`: Mission is generated by the planner but not yet routed.
- `ROUTED`: Mapped to a capability in the registry, validated, and ready for queue ingestion.
- `QUEUED`: Placed in the queue, waiting for an idle worker.
- `CLAIMED_LOCAL`: Claimed by a local simulated worker.
- `RUNNING_LOCAL`: Active mock execution on the worker thread.
- `COMPLETED_LOCAL_SIMULATION`: Successfully simulated, results generated.
- `BLOCKED_REAL_WORLD_SIDE_EFFECT`: Blocked because the capability requires real-world side effects (e.g. merge to master, spend money).
- `FAILED_RETRYABLE`: Simulated run failed but has retry attempts remaining.
- `FAILED_BLOCKED`: Retries exhausted; mission is blocked in a failed state.

## Worker Simulation States

Simulated worker threads follow a state machine:

- `IDLE`: Worker is waiting for a queued task.
- `CLAIMING`: Attempting to grab a mission from the queue.
- `RUNNING`: Emulating capability tasks.
- `REPORTING`: Writing execution artifacts and status.
- `DONE`: Completed execution cycle.
- `FAILED`: Encountered a simulated exception.

## Retry and Backoff Behavior

When a simulated mission fails, the router schedules it for retry:
- It increments the mission's `retry_count`.
- If `retry_count <= max_retry_attempts`, the mission transitions back to `QUEUED` (after a simulated delay).
- If retries are exhausted, it transitions to `FAILED_BLOCKED`.

## Result Artifact Format

Each completed simulated mission writes a mock artifact to indicate successful completion. The result artifact includes:
- `mission_id`
- `status`: `success` or `blocked`
- `output_file`
- `simulated_by_worker`

## Plugging in Real Workers Later

The local mission queue is designed to be fully modular:
- Currently, the router directly invokes `simulate-workers` in-memory.
- In the future, this in-memory list will be replaced by a database-backed Mission Queue table.
- Real autonomous worker agents (e.g., node scripts running on separate nodes) will poll this queue using active locks, claim entries, and execute real tasks while adhering to the same state model.
